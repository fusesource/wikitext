Getting Started

Workspace Setup

- Building Mylar feature from HEAD requires Eclipse SDK 3.2M4 and JDK 1.5. 

- To get the Mylar source code, in the CVS Repositories view, add the following CVS location: HOST = dev.eclipse.org  Repository Path: /home/technology.
If you do not have an account you may enter anonymous for the user name and leave the password blank.

- Check out all of the projects under org.eclipse.mylar (except mylar.aspectj, mylar.sandbox, mylar.viz, and mylar.zest which have additional dependencies).

Building from HEAD

- Associate the J2SE-1.5 with your installed JRE on the Execution Environments preference page.

Building the Eclipse 3.1 Branch

- Right-click the org.eclipse.mylar.java, mylar.ide, and mylar.xml project, Team -> Switch to Another Branch..., then Refresh the Branches and select e_3_1

The Initial Test

- Now that you have the source, the first thing to do is confirm that it all compiles without failing any of the unit tests. In the org.eclipse.mylar.tests project there is a test suite class (org.eclipse.mylar.tests.AllTests) that aggregates all tests in the Mylar project. Note that this test must be run as a JUnit Plug-in Test. Execution of this test suite should result in a GREEN bar (no failures) and there should not be any output in the console view as a result of running these tests. The HEAD repository is maintained so that it always passes this test suite.


Writing Patches   

- Write a failing unit test first (see Writing Tests below).

- Try to minimize the amount of changes to existing code.

- Synchronize when patches have been applied so that you don't have a merge/synchronize problem. Using the Synchronize view in Incoming Mode , you can update your local copy of the source with the most recent changes to the Mylar source repository. Synchronize frequently when working and prior to submitting a patch.

 

Writing Tests

- Mylar is maintained using test driven development. When writing patches, each added feature in the patch must be accompanied by a test case.
You will notice that most packages in the Mylar code base have an associated test package. For example, the unit tests for the org.eclipse.mylar.tasklist package are located in the
org.eclipse.mylar.tasklist.tests package. Within these 'tests' packages you will find a single test suite class that should reference all unit tests within that package. In the case of our chosen example the class AllTasklistTests.java does the job. If a new unit test is added to this package, a reference to the new unit test must be added to AllTasklistTests.java.

- Where unit tests already cover code related to your own, try to add test cases to these existing unit tests rather than creating entirely new classes.

- Mylar Active Plug-in Suite is a run environment for executing all unit tests currently within your active Mylar task context. By setting one of these run environments you can quickly execute whatever tests you are currently working on.

 

Submitting Patches

- Before patching check the Synchronize view in OUTGOING mode (press Change Set button to enable change set mode) to see what you have added/changed.
Look over you code in this view for extraneous debug code such as System.out.println etc. [Note that by right-clicking on the change set you can create patches from this view.]

- Eliminate all problems (Problems view) before submitting.

- The org.eclipse.mylar.tests.AllTests test suite must pass before submitting the patch.

- When using Bugzilla, open the bug report and submit the patch using the "Create New Attachment" link. Specify the patch file and denote it as a patch. Add any pertinent comments.
